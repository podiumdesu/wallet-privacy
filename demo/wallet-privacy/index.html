Technical details (optional)

How this test works behind the scenes

This deanonymization attack exploits how wallet permissions are scoped to website origins, combined with the widespread use of shared third-party trackers.

  1. When a user connects their wallet to a dApp (for example, dAppX.com), the wallet grants that origin permission to access the user’s address. The permission is scoped to the origin, not to a specific page or context.
  2. A third-party tracker is embedded on many websites, including Web3 dApps (such as dAppX.com) and ordinary Web2 sites (such as websiteA.com). dApps typically embed such trackers for benign purposes such as analytics or usage measurement.
  3. When the user later visits websiteA.com, the same tracker may also be present there. On this site, the tracker cannot directly access any wallet address, because the user never connected their wallet to websiteA.com.
  4. However, the tracker can invisibly load any dApp that embeds it inside a hidden <iframe> (Figure 1, Step 1 inject). The dApp does not know that it is being loaded in this way and does not cooperate with the attack.
  5. Inside the iframe, the page origin is dAppX.com. If a wallet injects its provider into cross-origin iframe contexts(Figure 1, Step 2 expose), the provider becomes available inside this iframe. As a result, the tracker script running there (under the dAppX.com origin) can call eth_accounts and obtain the wallet address that was previously authorized for dAppX.com.
    Wallets that do not inject providers into cross-origin iframes do not expose addresses at this step.
  6. The tracker then sends this address back to the top-level page (for the tracker on websiteA.com), where this address can be linked with other identifiers visible on websiteA.com (such as Google logins). (Figure 1, Step 3 link)

The tracker can inject many different dApps that embeds it (dAppX.com, dAppY.com, dAppZ.com, …) and succeeds whenever the user has previously connected their wallet to any one of them.

This enables silent Web2 ↔ Web3 identity linkage without user interaction, without cookies, and without cooperation or awareness from the dApps involved.
